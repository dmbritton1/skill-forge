# SkillForge v0.2 — Slice B Design: Retrieval & Tiering

*Approved design from brainstorming, 2026-07-10. Parent spec: skillforge-architecture-v4.md (draft 0.7), Section 8. Builds on Slice A (`2026-07-10-v0.2-slice-a-design.md`): ledger, trust registry, trust-gated sync.*

## Slice B scope

In: BM25 retrieval (`scripts/retrieve.py`), retrieval index + hot-budget tiering (extending `scripts/sync.py`), UserPromptSubmit injection hook, `/skillforge:find` command, injection logging to the ledger.

Out (deferred): confidence buckets and hedged candidate wording (Slice C), symptom-triggered anti-skill injection (Slice C), precondition checks against repo deps (Slice C staleness work), ε-holdouts (v0.3), embedding retrieval (only if BM25 outgrows keywords, per parent spec §8).

## Design decisions made during brainstorming

1. **Real tiering now, interim rank.** The parent spec gates hot-tier eligibility on the `trusted` confidence bucket, which doesn't exist until Slice C. Rather than shipping dead retrieval code (warm tier empty while everything stays natively materialized) or deferring the budget, Slice B implements the full hot/warm split with an interim ranking — ledger `injections + uses` descending, tiebreak most-recent `save` event, then name — explicitly documented as a placeholder that Slice C's buckets replace. Tier assignment lives in `sync.py`, where materialization already lives.
2. **No double-fire by construction.** A skill is native XOR retrievable: hot skills are materialized and excluded from retrieval; warm skills are in the index and never materialized. The retriever filters on the index's `tier` field.
3. **Word-noise control is two-layer.** BM25's IDF weighting makes common-word overlap nearly worthless; a ≥2-distinct-matched-terms gate refuses to inject on the strength of any single matched term, killing the single-rare-word false-positive that small-library BM25 can't distinguish from intent. `/find` and the hot tier cover the rare legitimate one-word case.
4. **The index is derived state, like native dirs.** Compiled by sync (which already walks stores and knows the trusted set) at every SessionStart and save. Quarantined content never enters the index — the trust gate applies at compile time, not query time.

## Components

### 1. Retrieval index — `~/.claude/skillforge/index.json`, compiled by `sync.py`

One JSON file covering both stores as of the last sync:

```json
{
  "compiled_ts": "2026-07-10T20:00:00+00:00",
  "hot_budget_tokens": 1500,
  "entries": [
    {
      "name": "stripe-webhook-integration",
      "kind": "skill",
      "scope": "global",
      "root": "/Users/x",                    // base dir whose store holds it
      "description": "Set up a Stripe webhook endpoint...",
      "tier": "hot",
      "est_tokens": 412,                      // body estimate, len/4
      "path": "/Users/x/.claude/skillforge/skills/stripe-webhook-integration/SKILL.md"
    }
  ]
}
```

- Trusted skills only. Descriptions come from frontmatter (folded scalars joined).
- Rebuilt wholesale on every `sync()` run; no incremental updates.
- Project-scoped entries appear when sync ran with that `--project-root`; `root` records which base they belong to.

### 2. Hot budget — in `sync.py`

- Rank all trusted skills: `(injections + uses)` from `skill_aggregates` DESC, then most-recent `save` event ts DESC, then name ASC. *(Interim formula — replaced by Slice C buckets.)*
- Estimate description tokens as `len(description) / 4`.
- Walk the ranking, materializing natively until the cumulative description-token budget (default **1500**, env `SKILLFORGE_HOT_BUDGET` for tests) is spent; every remaining skill is `tier: warm` and its native copy (if any) is evicted.
- Anti-skills compete in the same ranking in Slice B; their proper channel (symptom triggering) arrives in C.
- Existing sync semantics unchanged: quarantined/modified/orphaned still evicted; hashed bytes still equal materialized bytes.

### 3. Retriever — `scripts/retrieve.py`, UserPromptSubmit hook

Hook stdin (JSON from Claude Code) supplies `session_id`, `prompt`, `cwd`. Flow:

1. Load `index.json`; consider only `tier: warm` entries whose `root` is `Path.home()` or a prefix of `cwd` (project skills only fire in their repo).
2. Tokenize prompt and descriptions: lowercase, split on non-alphanumeric, drop tokens < 3 chars and pure numbers.
3. BM25 (k1=1.5, b=0.75) over `name + description` per entry.
4. Gates, in order:
   - score > 0 AND ≥ 2 distinct query terms matched the entry
   - session dedupe: skip names already in `~/.claude/skillforge/state/session-<session_id>.json`
   - max 3 `kind: skill` entries; anti-skills exempt from the count cap
   - cumulative injection budget 1200 tokens (est. `len(body)/4`) over full SKILL.md bodies, in score order; an entry that would exceed the budget is skipped (not truncated)
5. Emit hook JSON: `{"hookSpecificOutput": {"hookEventName": "UserPromptSubmit", "additionalContext": "..."}}` — each skill framed as `--- SkillForge retrieved skill '<name>' (apply if relevant): ---` followed by its full SKILL.md content.
6. Record injected names in the session state file; log one ledger event per injection (`event_type=injection, tier=warm, trigger=prompt, session=<id>`).
7. Entire main wrapped in try/except → exit 0 always, empty output on any failure. A broken index means no injection, never a broken session.

Latency: index load + BM25 over ≤ a few hundred short descriptions, all local files — well under the parent spec's 200ms budget.

CLI modes: hook mode (default, reads stdin JSON) and `--search "<topic>" [--limit N]` (Section 4).

### 4. `/skillforge:find` — `commands/find.md`

Cold-path pull: runs `retrieve.py --search "<topic>"` — no gates, no budget, no dedupe, hot and warm alike — printing top 10 as `name | kind | tier | scope | description | path`. The command instructs the model to present results and offer to read any hit. Quarantined skills excluded (they're not in the index).

### 5. Housekeeping

- `sync.py` opportunistically deletes `state/session-*.json` files older than 7 days.
- `hooks/hooks.json` gains the UserPromptSubmit entry alongside SessionStart:
  `python3 "${CLAUDE_PLUGIN_ROOT}/scripts/retrieve.py"` (hook mode reads stdin; `--project-root` not needed — cwd comes from hook input).

## Data flow (one prompt)

```
user prompt → UserPromptSubmit hook → retrieve.py
  index.json (trusted, warm, root-scoped) → BM25 → gates → additionalContext
  → session state updated → ledger injection events
model sees: native hot skills (standing) + injected warm skills (this prompt)
```

## Error handling

- Missing/corrupt `index.json` → retriever emits nothing, exit 0.
- Missing state dir → created on demand; unreadable state file → treated as empty (dedupe resets, harmless).
- Ledger write failure → injection still emitted (context first, bookkeeping second), error to stderr.
- Sync failures already fail-safe from Slice A (exit 0, quarantine-by-default).

## Testing

Stdlib-assert files, sandbox-HOME pattern:

- `tests/test_retrieve.py` — BM25 ranks the on-topic skill first over realistic fixtures; single-matched-term entry rejected by the two-term gate; count cap (3 skills) with anti-skill exemption; 1200-token budget skips oversized entries; session dedupe across two calls; corrupt index → empty output, exit 0; `--search` returns hot + warm.
- `tests/test_sync.py` additions — budget split (high-use skill hot, overflow warm + native evicted, `SKILLFORGE_HOT_BUDGET` shrunk for the test); index compiled with trusted-only entries and correct tiers; stale session-state cleanup.
- E2E — sandbox HOME: save two skills, shrink budget, verify tier split in index + native dirs; pipe fake hook stdin through `retrieve.py`, verify additionalContext, ledger row, and dedupe on a second identical prompt.

## Non-goals / notes

- With the library at ~0 skills today, everything fits hot and the warm path idles — this slice is the infrastructure that makes the library safe to grow; the mid-work payoff loop starts here.
- Injection wording carries no confidence hedges yet (no buckets until C).
- BM25 constants (k1=1.5, b=0.75) and gates (2 terms, 3 skills, 1200 tokens) are the parent spec's defaults, centralized as constants in `retrieve.py`.
